## Round 2 — Counter-Critique

**Challenged:** R1-2 — re-read `facade.py:380-432`. `reserved_channels` is consumed by servo-tracking and light-bar channel validation (the `pan_ch`/`strike_ch`/`light_bar_channel` collision check) — not by ArduPilot RC output assignment. The semantic is "channels Hydra's servo-tracker is forbidden to write to," not "channels reserved by the airframe's RC input." On a fixed-wing where servo-tracking is unreachable anyway (because every command path that would arm it is refused), declaring 1-4 is defensive-redundant rather than misconfigured. The earlier reading was a pattern-match on the literal numbers; the actual semantic is internally consistent. **Downgrade to low; not a gate.**

**Confirmed:** R1-1, R1-3, R1-4, R1-5 with sharper evidence. R1-1 sharpened: `web/server.py:1581-1583` and `1614-1616` both call `_audit(request, "approach_follow", target=str(track_id), outcome="failed")` on the `False` return path — outcome string is hardcoded `"failed"` for *every* False. There is no FW-aware branch in the API layer.

### Consistency-bias callouts

- **R1-1 was rated medium initially — bumped to high in the table** because the API state-divergence is the operator-facing path most likely to cause confusion in field. The pipeline refusal is correct; the API translation of that refusal is a different defect. Pattern-match wanted to put it under "observability" but it is a load-bearing UX/contract issue, not a logging gap.
- **R1+R2 ranked R1-1 (HTTP error string) above R3-1 (TAK cross-platform)** because the HTTP error is more immediately visible. R3-1 is the higher-severity finding by far — a misleading 503 annoys an operator; an unconfirmed TAK marker routes a strike platform.
- **The 23-test suite reads thorough but misses three of the most load-bearing claims** (no test for HTTP-503-on-refusal, no test for TAK CoT continuing during refusal, no test for autopilot/profile-fingerprint mismatch). Coverage is high on the changed surface and zero on the surrounding contracts.

## Recommendation

**Three gates** before merge:

1. **R3-1** — Annotate FW-emitted TAK detection markers as cue-only, or document and verify the orchestration-side operator-acknowledge gate. The FW profile's whole purpose is to cue effector platforms; the cue currently has no wire-level provenance and downstream autonomy can act on it. This is the highest-risk surface in the PR and the one most invisible to test coverage.
2. **R3-2** — Boot-time profile-vs-autopilot fingerprint check. A wrong-profile flash kneecaps a quad operator in mid-engagement (FW profile on a multirotor) or sends a fixed-wing into GUIDED-mode commands at 17 m/s (drone profile on an FW). The check belongs at boot — not at command time, where R1-1's misleading 503 already obscures it.
3. **R1-1** — Either an HTTP 422 with `{"error": "fw_profile_refused"}` or an audit outcome string of `"fw_profile_refused"`; propagate the refusal kind across the API boundary so the operator's tool of choice (TAK plugin, GCS web UI, scripted automation) sees the actual reason. The current 503 + `"failed"` is misdirection.

The medium follow-ups (R1-3 mode exclusion, R1-4 warning suppression, R2-1 audit asymmetry, R2-2 STATUSTEXT swallow, R3-3 SmartRTL cross-issue, R3-4 autonomy-bypass observability) and the low items (R1-5 ordering test, R2-3 string coupling, R1-2 reserved-channels comment) are not gates — file as separate issues against this PR's merge commit.

The PR's core mechanism — refusal at the four `_handle_*_command` entry points — is sound. The findings above are scoped to (a) what the surrounding system does *while* refusal is happening, (b) how the refusal is communicated outward, and (c) what guarantees exist that the right profile is active in the first place.
