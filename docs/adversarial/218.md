## Round 2 — Counter-Critique

**Challenged:** R1-1 — re-read the lock. `requirements.lock:10` shows `pymavlink==2.4.49`, and the container build path on the Jetson uses the lock (`pip install -r requirements.lock`), not the open floor. Production has been on 2.4.49 since the lock was last regenerated; this PR adjusts the floor in `requirements.txt` to match what the lock has been doing. **Runtime delta is zero on the supported install path. Downgrade R1-1 to low — it stays on the register only because the unlocked install path exists.**

**Confirmed and sharpened:** R1-2 is now the load-bearing finding. The `_instances is None` workaround was written when ArduPilot/pymavlink had inconsistent instance bookkeeping; the 2.4.49 changelog says that area was repaired. Two possibilities:

1. The TypeError no longer fires from `_instances`, so the `except` block catches only future, unrelated TypeErrors — silently. This is a defensive-code-rot pattern: catches stay around long after their original cause is fixed and start eating bugs the maintainer doesn't know about.
2. The workaround is still needed (the upstream fix was for DFReader log-replay, not live-stream message bookkeeping; mavlink_io.py is on the live-stream path).

The PR does not resolve which is true. The minimum-effort answer is to keep the swallow but log the exception once-per-process before continuing, so the next maintainer can tell whether it ever fires post-2.4.49.
