## Round 1 -- Pattern Match

4 findings. Dominant pattern: a parallel auth surface (`_web_password` plaintext vs `_web_password_hash`) where the precedence rules are obvious from the docstring but not from the runtime behavior alone.

| ID | Sev | Cat | Where | One-line claim |
|---|---|---|---|---|
| R1-1 | high | auth-bypass-risk | server.py:configure_web_password (hash branch) | When `password_hash` is passed, `_web_password` is reset to None. Good. But if a future caller passes BOTH a non-empty plaintext password AND a non-empty hash, the hash wins and the plaintext is silently discarded. An operator who explicitly set `[web].web_password = "hunter2"` for a dev override would see their override silently overridden by an inherited `[identity].web_password_hash`. Documented in docstring but not enforced. |
| R1-2 | medium | legacy-fallback-divergence | facade.py:`effective_token = identity_token or api_token` | The identity-token branch SKIPS the auto-generation of `[web].api_token`. If the operator later unsets `[identity]` (manual edit) but never reruns Platform Setup, the unit boots with NO token, no auto-gen, no fallback. The facade currently sets `api_token = ""` in that path (line: identity_token=non-empty -> skip-autogen -> effective_token=identity_token); a partial unset of `[identity]` post-boot leaves the unit unauthenticated. |
| R1-3 | medium | factory-default-blast-radius | config.ini.factory (require_auth_for_control = true) | Existing dev workstations that pull this branch and `restore_factory()` will suddenly require an api_token where they did not before. Pre-existing dev workflows (curl /api/abort, manual config POSTs) break. The audit log will show 401 spam from any tooling that did not previously authenticate. Not a security defect -- the OPPOSITE of one -- but a behavior change worth calling out in CHANGELOG. |
| R1-4 | low | wipe-idempotency | scripts/platform_clone_wipe.sh | Running the wipe twice in sequence is safe (each clear is destructive but bounded). Running it on a fresh-clone unit (no [identity] set yet) is also safe. Running it on a unit where `[identity]` is partially populated (write_identity_to_config crashed mid-write) is also safe because configparser silently accepts missing keys. No defect; worth pinning in a test. |

## Round 2 -- Counter-Critique

**Confirmed R1-1, R1-2** with sharper evidence. **R1-3** stands as-is (operator-visible behavior change, document in PR body).

- R1-1: The docstring says "exactly one of password or password_hash should be set ... if both, password_hash wins." That is the contract. The risk is not a security regression (constant-time comparison is correct on both paths via `_verify_web_password` -> `hmac.compare_digest` for plaintext or `identity.verify_password` for hash) but an operational surprise. **Downgrade R1-1 to medium**: the implementation matches the docstring, but the precedence rule is non-obvious from `server.py` alone. Add a one-line WARNING when both paths arrive non-empty.
- R1-2: re-read `facade.py` lines 1180-1240. If `identity_token` is set and `[web].api_token` was previously auto-generated (now stale), the stale token sits in `[web].api_token` on disk but is NOT used at runtime (effective_token uses identity_token). That is correct. The risk surfaces only on a SUBSEQUENT boot after manual `[identity]` removal -- at that point the unit has no token and `require_auth_for_control=true` from the factory file means /api/abort etc. are inaccessible. The unit is effectively bricked from a dashboard perspective until the operator either rerun Platform Setup or manually re-populates `[web].api_token`. **Acceptable** because: (a) it's an operator action that requires SSH access anyway, (b) the alternative -- auto-regenerating a token silently -- is worse (operator thinks Platform Setup didn't run).

R2 second-order findings:

| ID | Sev | Cat | Where | One-line claim |
|---|---|---|---|---|
| R2-1 | medium | session-cookie-secret-rotation | server.py:`_session_secret = secrets.token_bytes(32)` | The session cookie HMAC secret is regenerated at module import time. A pipeline restart invalidates all existing session cookies -- which is FINE for the hash-auth path (operator re-logs in with the password) but is operator-surprise when paired with the factory-flip in R1-3. Worth documenting "all existing dashboard sessions will be invalidated after first boot of an upgraded unit." |
| R2-2 | low | clone-wipe-no-machine-id-regen | scripts/platform_clone_wipe.sh | We `truncate -s 0 /etc/machine-id` but do NOT explicitly invoke `systemd-machine-id-setup --commit`. systemd regenerates the file on the next boot of the imaged unit anyway, but an operator running the wipe on a live system (not just pre-image) would have a zero-byte machine-id until reboot. Some tools (dbus) misbehave with an empty machine-id. Add a comment that wipe ASSUMES a poweroff-then-image flow. |

## Round 3 -- Orthogonal Sweep

**Shared framing of R1+R2:** *Both rounds audited the new branching logic in isolation -- "is the hash path correct, does the legacy path fall through correctly." Neither round asked (a) what the operator SEES on the dashboard when [identity] gets wired in for the first time, (b) whether the audit log captures the bridge activation (so we can post-hoc verify a take-home unit actually routed through [identity]), or (c) how this interacts with the SETTINGS / Platform Setup wizard UIs -- do any of the existing UI surfaces need to reflect the hash-path activation?*

R3 findings:

| ID | Sev | Cat | Where | One-line claim |
|---|---|---|---|---|
| **R3-1** | medium | dashboard-divergence | server.py + auth_status endpoint | The `/auth/status` endpoint returns `{password_enabled: bool, authenticated: bool}` but does NOT distinguish "password is hash-backed from Platform Setup" vs "password is plaintext legacy". An operator looking at the dashboard cannot tell whether Platform Setup ran correctly without SSHing into the unit and grep'ing config.ini. Add an `auth_source: "identity" \| "legacy" \| "disabled"` field to /auth/status. |
| R3-2 | medium | audit-log-coverage | facade.py:identity-bridge log lines | The facade logs `API auth routed via [identity].api_token` and `Web password auth routed via [identity].web_password_hash` at INFO level. These go to standard logs, NOT to the audit log. Post-incident replay of "did this unit's Platform Setup actually take effect?" requires grepping hydra.log, not the audit JSONL. Worth emitting an `IDENTITY_BRIDGE_ACTIVE` audit row at startup. |
| R3-3 | low | settings-ui-stale | hydra_detect/web/templates/settings.html | The Settings page (settings.html) likely exposes a "Web Password" input field tied to `[web].web_password`. When `[identity].web_password_hash` is set, editing that field in the UI has no runtime effect (the bridge ignores the plaintext path). UI shows a value, save succeeds, login still requires the OLD identity password. The operator's only signal is a failed login attempt. Doc-only for now; a real fix is to hide/grey-out the Web Password field when `[identity]` is populated. |

### Consistency-bias callouts

- **R1+R2 focused on the auth-routing branching logic; R3 caught the dashboard/audit-surface gap** -- same framing-break as PR #240's R3-1 (pan_disabled exposed but not rendered). This is the third time in a row this framing-break has surfaced. The pattern: new state fields land in `get_status()` / `/auth/status` / etc., but the dashboard widgets that render them are not updated. Consider a "dashboard widget audit" CI gate that flags new keys in those endpoints without a corresponding template diff.
- **R3-3 is the most operator-visible failure mode** in this PR. An operator trying to "change my web password" via Settings will silently fail. Either hide the field or surface an info banner.

## Round 3 explicit framing-break

The framing both prior rounds shared was: *"check that the new auth-routing branching is internally correct and that the legacy fallback still works for un-set-up units."* The framing-break in R3 was asking: *"how does an operator, looking only at the dashboard, KNOW that Platform Setup ran and the identity bridge is active?"* That's how R3-1 (no auth_source field), R3-2 (no audit row for bridge activation), and R3-3 (Settings UI silently no-ops) all surfaced -- none of them are bugs in the new code. All of them are operationally-visible gaps the new code does not address.
