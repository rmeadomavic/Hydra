## Round 2 — Counter-Critique

**Challenged R1-1:** re-read `hydra_detect/audit/__init__.py` and `audit_log.py`. The audit module imports stdlib only (`configparser`, `json`, `logging`, `threading`, `os`, `time`, `collections.deque`, `pathlib.Path`) and the module-level work is `_default_sink = AuditSink()` + `_attach_lock = threading.Lock()`. `attach_to_logger()` itself is idempotent and does not touch disk. The future-failure scenario in R1-1 is hypothetical and explicitly addressed by the existing `_attach_lock`. **R1-1 downgraded to LOW; tracked as a code-review checklist item for future audit work, not a gate.**

**Confirmed R1-2 at MEDIUM.** Verified by reading the existing `write_config` — it acquires `fcntl.flock(lock_fd, fcntl.LOCK_EX)` on the lock_fd opened against the TARGET path, not the `.tmp`. A concurrent recovery sweeping `.tmp` while another process holds the EX lock on the target would still race because the sweep does not take the lock. The mitigation is documented (single-writer assumption is the operating contract) but the comment in this PR could be louder. Tracked as a follow-up doc tightening.

**Confirmed R1-3 at LOW.** Same exposure baseline as the existing classifier. The mitigation is to use `kind="recovery_from_bak"` directly via `AuditSink.push()` instead of relying on message-text classification. Tracked but not blocking — the project-wide convention is logger-message-based classification, changing it for this one emitter would create inconsistency.

| R2-1 | medium | deployed-stale-lkg | hydra_detect/config_lkg.py removal vs. existing units | Deployed Jetson units that have already taken an `.lkg` snapshot under a prior Hydra release will keep the file on disk after this update. The removal leaves `.lkg` files orphaned alongside `config.ini`. Operators following stale runbooks ("restore from LKG") will look for the file, find it, and try to copy it manually — and they'll succeed, getting a config that the running code has no knowledge of. | R2-2 | low | ring-vs-file-sink-drift | hydra_detect/__main__.py recovery event ordering | The recovery event lands in the in-memory ring but NOT in the durable JSONL file sink for THIS boot (file sink attaches after pipeline init, which is after recovery). The implication: a unit that boots from `.bak`, runs for an hour, then crashes before any new audit event fires loses the durable record of the recovery. Ring is bounded (500 events) so it also rotates out. Documented in PR body — could be lifted by attaching a temporary file sink to a recovery-only path. Out of scope for this PR. |

Order is correct (specific-before-general) but adding new emitters in front of existing ones is a maintenance smell — future engineers may add new branches at the top and inadvertently shadow downstream rules. A comment explaining the order convention would help. Cosmetic. |

Out of scope to add a process-level lock — that is a separate infrastructure concern.
