## Round 2 — Counter-Critique

**Confirmed R1-1.** Re-reading the design — the global state IS the gate. A registry-per-request lookup would re-walk all 20 evaluators on every mission-start hit; the 2 s cache is exactly what #245 introduced to avoid that. Fail-open on first boot is consistent with #245's conservative-by-default posture (evaluators that cannot read their signal return WARN, not BLOCKED). Pin the contract in `is_disk_blocked()` docstring.

**Confirmed R1-2 with sharper framing.** A headless / API-driven deployment (Hydra in a CI smoke test, a phone-home consumer that polls only `/api/health`) would never refresh the gate. The fix is either (a) a background refresh thread or (b) require the dashboard poll for the gate to be live. (a) duplicates the cache machinery; (b) is acceptable for the spec but should be documented in the PR body.

**Confirmed R1-3.** Two-section overlap is real. The boundary is: `[storage]` controls the cleanup script + boot-time check; `[capability.disk]` controls what the operator sees on the dashboard + mission-start gate. The implementation deliberately falls back from `[capability.disk]` to `[storage]` so existing configs keep working untouched (backward compat per #226 acceptance). New operators should prefer `[capability.disk]`.

**Confirmed R1-4 but downgraded.** Not material — the worst case is one extra crop landing after BLOCKED tripped, and the queue depth is bounded.

## Round 3 — Orthogonal Sweep
